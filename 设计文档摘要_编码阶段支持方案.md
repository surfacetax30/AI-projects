## 一、项目定位

**一句话：** 用 AI Agent 自动化汽车零部件 OTS (Off-Tool Sample) 认可流程中的报告解析、资料完整性检查和审批辅助，减少 PE 工程师 80% 的重复工作。

---

## 二、业务流程（6 节点 + 返工回路）

```
① 认可发起 ─→ ② 测试申请 ─→ ③ 报告收取 ─→ ④ 资料整理 ─→ ⑤ 认可会签 ─→ ⑥ 结案归档
                              ↑                                               |
                              └──── 返工完成 ──── 返工重测 ←── 异常发现 ←──────┘
```

| 节点 | 核心工作 | AI 介入程度 |
|------|---------|:----------:|
| ① 认可发起 | PE 录入零件信息，AI 推荐测试类型和资料清单 | 辅助 |
| ② 测试申请 | 自动填充模板，AI 生成申请描述，PE 确认后邮件发送 | 辅助 |
| ③ 报告收取 ⭐ | 邮件自动监听 → 多模态解析 PDF/图片/Excel → 字段提取 + 置信度校验 | **全自动** |
| ④ 资料整理 ⭐ | 动态清单核对，规则 + LLM 语义等价检查，缺失提醒/超时升级 | **全自动** |
| ⑤ 认可会签 | AI 生成风险摘要，多方审批流转，IM 催办 | 辅助 |
| ⑥ 结案归档 | 案例入库，向量知识库，纠错记录→SFT 数据集 | 全自动 |

## 三、V2.0 架构（5 层）

```
┌──────────────────────────────────────────────────────────────────┐
│  用户层：PE / 测试工程师 / 供应商 / 会签人员 / Web工作台 / IM      │
├──────────────────────────────────────────────────────────────────┤
│  编排层：状态机 + 路由决策表 + 规则引擎 + 10个人工确认节点 + 超时监控 │
│        【只有编排 Agent 有权改状态，功能 Agent 只产出数据+事件】     │
├──────────────────────────────────────────────────────────────────┤
│  Agent层：                                                       │
│  📧 邮件网关 Agent (IO密集型,不调LLM)                              │
│  → 🔍 解析 Agent (LLM密集型, Qwen3-VL-32B, GPU约束)                │
│  → 📋 资料 Agent (混合型, 规则+LLM+RAG)                            │
│  → 📄 报告 Agent (AI型, MVP暂缺)                                  │
├──────────────────────────────────────────────────────────────────┤
│  事件层：asyncio.Queue (MVP) → RabbitMQ (通知量>1000/天时升级)     │
├──────────────────────────────────────────────────────────────────┤
│  数据层：PostgreSQL / MinIO / Milvus-PGVector / LLM(Qwen3本地部署) │
│  部署形态：单 FastAPI 进程 + Docker Compose                        │
└──────────────────────────────────────────────────────────────────┘
```

## 四、状态机（8 状态）

```
CREATED → TEST_APPLYING → TESTING → REPORT_COLLECTING
                                          ↓
                                    DATA_ORGANIZING
                                          ↓
                                       SIGNING
                                          ↓
                                       CLOSED

分支：DATA_ORGANIZING 可触发 REWORKING → 回 REPORT_COLLECTING
      SIGNING 驳回 → 回 DATA_ORGANIZING
      HUMAN_PENDING 是 flag（不是独立状态），挂载在任意主状态上
```

## 五、MVP 实施范围

### ✅ 实施（1 编排 + 3 功能 Agent）

| Agent | 类型 | 技术栈 | 核心输出 |
|-------|------|--------|---------|
| **编排 Agent** | 状态机+规则表 | Python, 不调LLM | 状态流转、路由决策 |
| **邮件网关 Agent** | IO密集型 | IMAPClient + asyncio | 邮件监听、附件下载、规则过滤 |
| **解析 Agent** | LLM密集型 | Qwen3-VL-32B | 字段提取、置信度评分、异常检测 |
| **资料 Agent** | 混合型 | 规则引擎 + LLM + pgvector | 清单核对、语义等价、缺失提醒 |

### ❌ 暂不实施

- 报告 Agent（报告自动生成、FAIL 分析、返工管理）→ V1.5
- PLM/ERP 集成
- 供应商协作门户
- SFT 模型微调
- 移动端

## 六、关键设计决策（编码时必须遵守）

### 6.1 通信协议
- 所有 Agent 通过 `asyncio.Queue` 事件总线通信，**13 种标准事件类型**
- **只有编排 Agent 能改任务状态**，功能 Agent 只产出数据 + 发布事件
- 通知消费者是一个独立 asyncio 协程

### 6.2 置信度机制（固定阈值）

| 场景 | 阈值 | 低于阈值行为 |
|------|------|------------|
| 字段提取 | ≥ 0.85 | HUMAN_PENDING flag=true |
| 数值提取 | ≥ 0.90 | 标记待审，高亮 |
| 分类 | ≥ 0.80 | 提供 Top-3 候选 |
| 语义等价 | ≥ 0.85 | 标记待审 |
| 异常检测 | ≥ 0.80 | 通知但不阻塞 |

### 6.3 LLM 三层纠错
1. **L1 格式修复**：JSON Schema 校验失败 → 注入格式约束重试（最多 2 次）
2. **L2 语义修复**：值不合理 → 注入领域规则重试（最多 1 次）
3. **L3 降级**：标记低置信度，不阻塞流程，交人工确认

累计 3 次失败 → 终止自动处理，写入异常队列，等待人工介入。

### 6.4 人工确认节点（10 个）
H1: 低置信度字段 | H2: 分类不确定 | H3: 异常检测 | H4: 语义等价不确定 | H5: 超差判定 | H6: 材料偏差 | H7: 供应商资质 | H8: 返工审核 | H9: 版本冲突 | H10: 弃用确认
