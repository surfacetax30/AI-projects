OTS-AHA v3 PRD：Agent 架构重构

本版本将产品重心从“自动解析测试报告”调整为 OTS 认可资料包智能处理助手：系统负责资料接入、文档证据化、 问题识别、认可就绪判断，并把带依据的建议交给 PE 完成人工确认。

项目OTS Approval Helping Agent
版本v3 Agent 架构重构
策略新增版本，不覆盖旧文档
日期2026-05-30

1. 产品定位

一句话目标： PE 每天处理 OTS 资料时，系统自动接入资料、理解文档内容、判断认可任务是否就绪， 识别测试文档中的问题，并基于规则与历史案例给出可追溯建议，最终由 PE 复核推进。

v3 的核心变化是从“单份报告解析工具”升级为“资料包级工作流助手”。 OTS 认可真实场景不是只判断一份测试报告，而是围绕一个零件任务处理多份文件、 多个来源、多种版本和多个确认节点。

因此，Agent 链路应围绕 资料进入 → 文档证据化 → 问题识别 → PE建议 → 人工确认/推进 设计，而不是围绕“邮件 → 解析 → 检查 → 报告”设计。

2. 用户与核心场景

角色 核心诉求 v3 产品承诺
PE / 产品工程师 快速判断资料是否齐、是否可信、下一步该做什么。 集中呈现任务就绪度、低置信字段、缺失项、版本冲突和待确认事项。
测试工程师 提交测试报告后能被系统正确识别并关联任务。 通过 Intake Agent 统一接入，避免邮箱规则误拦后静默丢失。
供应商 知道缺什么、为什么缺、如何补。 MVP 先由 PE 工作台承接，后续再扩展供应商门户。
会签人员 看到清晰的认可依据和风险提示。 只在任务达到认可就绪后进入会签，不让原始杂乱资料直接流入会签。
质量经理 关注一次通过率、异常漏检、处理周期和人工介入率。 以任务、文档、证据、确认项为基础沉淀可审计数据。

3. v3 核心业务链路

1. 资料进入 邮件、手动上传、Webhook 进入统一入口，生成可追踪的原始资料记录。
2. 文档证据化 对资料进行类型识别、文本提取、字段提取、置信度标注，形成证据。
3. 问题识别 从测试文档中识别 FAIL、BORDERLINE、版本不一致、标准缺失、数值异常。
4. 建议生成 基于证据、规则、历史案例，生成 PE 可采纳或驳回的处理建议。
5. 人工确认 低置信、冲突、建议动作、误拦恢复进入工作台，PE 复核后回写结果。
6. 任务推进 编排 Agent 根据状态机和事件表推进任务或触发补充资料。

4. 三层输出模型

文档理解 Agent 不再只输出字段。它先把文件转成证据和问题； 认可就绪度 Agent 再把问题放回任务上下文，结合规则和历史案例形成建议。

Evidence / 证据

报告原文、字段值、页码/表格位置、置信度、适用的零件与测试类型。

Issue / 问题

FAIL、BORDERLINE、版本冲突、测试标准缺失、数值异常、报告日期异常。

Recommendation / 建议

建议 PE 要求补资料、复测、整改说明、风险会签，或确认可进入下一步。

建议能力边界

RAG 历史案例只作为建议依据和参考，不作为自动审批依据。 系统必须展示参考案例、相似原因、适用条件和置信度，最终动作由 PE 确认。

5. Agent 职责边界

Host

Orchestrator Agent / 编排 Agent

职责：状态机、路由决策、人工确认节点、事件消费、任务推进。

输入：接入事件、证据事件、就绪度事件、人工确认事件。

输出：任务状态变更、ReviewItem 创建、通知触发、补充资料请求。

原则：不调用 LLM，不直接解析文件，只做确定性流程决策。

入口

Intake Agent / 资料接入 Agent

职责：承接邮件、上传、Webhook、未来供应商门户，完成过滤、去重、关联和入队。

输入：邮件元数据、附件、上传文件、外部推送。

输出：SourceDocument、接入分类、待人工接入确认项。

变化：从 MailGateway 升级为资料接入层，不再被邮箱形态限制。

理解

Document Understanding Agent / 文档理解 Agent

职责：文件解析、文档类型识别、字段提取、证据生成、测试问题识别、置信度评分。

输入：SourceDocument 与任务上下文。

输出：DocumentEvidence、DocumentIssue、低置信字段、疑似无关文件标记。

原则：输出证据和问题，不独自生成完整处置建议，不输出最终审批结论。

判断

Approval Readiness Agent / 认可就绪度 Agent

职责：按任务维度检查资料包完整性、版本一致性、关键报告结论、阻塞项和 PE 建议。

输入：任务上下文、清单模板、全部 DocumentEvidence、DocumentIssue、历史案例参考。

输出：ReadinessDecision、缺失项、冲突项、PERecommendation。

变化：从字段完整性检查升级为认可就绪判断和 PE 辅助建议。

Human Review Workbench / 人工复核工作台

Human Review Workbench 不是 Agent，但必须是一等产品模块。 它承接所有低置信、冲突、误拦恢复、缺失确认、版本选择、资料补充判断和建议采纳。 没有这个闭环，系统只能做自动化 demo，不能成为 PE 可依赖的工作台。

6. MVP 范围

MVP 必须做

  • 统一资料接入：邮件、上传、Webhook 进入同一接入层。
  • 文档证据化：识别文档类型、提取核心字段、输出置信度。
  • 问题识别：从测试文档中标记 FAIL、BORDERLINE、版本冲突、标准缺失。
  • 轻量建议：基于规则给出 PE 建议动作，RAG 历史案例先作为可选参考。
  • 人工复核闭环：所有不确定项和建议动作进入 ReviewItem，并支持 PE 确认。
  • 可追溯记录：任务、文档、证据、确认项、状态变更可回溯。

MVP 不做

  • 报告 Agent：自动生成认可报告草稿延后到 V1.5。
  • 独立 RAG Agent：历史案例先作为建议辅助能力，不拆成 Agent。
  • SFT：等累计足够纠错样本后再启动。
  • 供应商门户：MVP 由 PE 工作台承接补充资料动作。
  • 复杂返工根因分析：先记录阻塞项，不做自动根因推理。

7. 核心产品对象

  • ApprovalTask：一个零件 OTS 认可任务，是状态机和资料包的主对象。
  • SourceDocument：进入系统的一份原始资料，保留来源、文件、接入判断和追踪信息。
  • DocumentEvidence：文档理解后形成的结构化证据，包含字段、置信度、出处和适用范围。
  • DocumentIssue：从测试文档和资料包中识别的问题，包含类型、严重度、证据引用、影响范围和置信度。
  • PERecommendation：面向 PE 的建议动作，包含建议、理由、历史案例引用、风险等级和是否必须人工确认。
  • HistoricalCaseRef：RAG 召回的历史案例引用，必须包含来源、相似点、差异点、适用条件和相似度。
  • ReviewItem：需要 PE 人工确认的事项，例如低置信字段、版本冲突、资料误拦、缺失项确认。
  • ReadinessDecision：认可就绪判断结果，典型取值包括 ready_for_signing、need_supplement、need_human_review、blocked_by_conflict。

8. 成功指标

指标 MVP 目标 为什么重要
资料接入准确率 OTS 相关资料识别准确率 ≥ 90%，误拦可恢复率 100% 避免供应商报告被静默丢弃。
文档识别准确率 主流 OTS 文档类型识别准确率 ≥ 90% 后续证据提取和清单判断依赖文档类型。
核心字段提取准确率 真实报告核心字段 ≥ 90% 零件号、测试结论、版本号、日期等错误会造成业务风险。
认可就绪判断准确率 PE 抽检通过率 ≥ 85% 衡量系统是否真正理解“能不能进入下一步”。
测试问题识别召回率 真实报告中的关键 FAIL / BORDERLINE / 冲突问题召回率 ≥ 85% 漏掉问题比多提醒更危险，问题识别优先关注召回。
PE 建议采纳率 系统建议中 ≥ 60% 被 PE 采纳或小幅修改后采纳 衡量建议是否真的帮 PE 判断下一步，而不是生成漂亮废话。
人工复核命中率 进入 ReviewItem 的事项中 ≥ 70% 被 PE 认为值得复核 避免工作台变成噪音中心。
PE 节省时间 单任务资料初筛耗时降低 50% 以上 MVP 是否有用户价值的直接指标。

9. 风险与约束

审批权风险

LLM 不直接给出最终审批决定。它只能输出证据、置信度、冲突提示和建议。

建议越权风险

PERecommendation 必须呈现为“建议动作 + 依据 + 需确认”，不能呈现为“系统已决定”。

低置信风险

低置信字段、冲突资料、非标准报告必须进入人工复核，不允许静默放行。

评测偏差风险

真实报告评测优先于模拟 case。模拟 case 只能验证格式，不能证明业务可用。

范围膨胀风险

报告生成、独立 RAG Agent、SFT、供应商门户不能混入 MVP，否则核心闭环会被稀释。

10. 验收场景

场景 预期结果 验收关注点
收到一封带测试报告附件的 OTS 邮件 Intake 生成 SourceDocument，文档理解生成证据，就绪度判断更新任务。 邮件、附件、任务、证据是否能被串起来。
资料包缺少关键报告 ReadinessDecision 返回 need_supplement，并生成清晰缺失项。 PE 是否知道缺什么、向谁要、为什么要。
同一零件存在多个软件版本资料 生成 ReviewItem，要求 PE 选择有效版本或标记冲突。 版本冲突不能被系统静默覆盖。
测试报告中出现 FAIL 或 BORDERLINE 生成 DocumentIssue，并由 Approval Readiness 给出 PERecommendation。 建议必须说明理由、证据出处、历史案例参考和是否阻塞认可。
