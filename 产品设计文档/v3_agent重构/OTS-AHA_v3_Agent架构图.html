OTS-AHA v3 Agent 架构图

2. 各层职责

用户层

PE 工作台是 MVP 的主入口，负责任务查看、人工复核、资料补充判断和状态确认。供应商资料入口和会签人员视图作为后续扩展。

PE 工作台任务看板人工复核会签视图

编排层

Orchestrator Agent 负责确定性流程决策。它消费事件、读写任务状态、创建 ReviewItem、触发通知。

关键原则：编排 Agent 不调用 LLM，不解析文件，不生成业务证据。

Agent 层

功能 Agent 分别对应资料接入、文档理解、认可就绪判断。文档理解 Agent 可识别测试问题，但完整建议由认可就绪度 Agent 结合任务上下文生成。

Intake AgentDocument Understanding AgentApproval Readiness Agent

参考能力层

规则库、历史案例 RAG、建议生成器为认可就绪度判断提供依据。它们是参考能力，不是独立审批 Agent。

业务规则库HistoricalCaseRefPERecommendation

事件层

MVP 使用进程内事件总线，保留升级到 RabbitMQ 的路径。所有关键事件进入 EventLog / AuditLog，保证链路可追溯。

数据层

数据模型围绕任务、资料、证据、问题、建议、清单、确认项展开。它们是后续工程实现和评测的统一语言。

3. 核心原则卡片

编排 Agent 不调用 LLM

流程推进必须可解释、可复现、可审计。LLM 可以提供证据和建议，但不能决定任务状态。

功能 Agent 不直接改状态

Intake、Document Understanding、Approval Readiness 都只发布事件和结构化输出。

LLM 输出证据，不输出审批决定

LLM 的产品位置是“理解、问题识别与辅助建议”，不是替代 PE、质量、会签人员作最终决定。

RAG 只能作为参考依据

历史案例必须带来源、相似点、差异点和适用条件，不能被当成自动审批依据。

人工复核工作台是核心闭环

低置信和冲突不是系统失败，而是 L3 Agent 产品必须承接的正常工作流。

4. 节点化能力矩阵

Intake 识别资料来源、过滤无关文件、关联任务、保留误拦恢复入口。
Understanding 形成 DocumentEvidence，并识别 DocumentIssue。
Advisory 基于规则与 HistoricalCaseRef 形成 PERecommendation。
Review PE 对低置信、冲突、建议动作进行采纳、修正或驳回。

5. Agent 输入输出边界

模块 输入 输出 不负责
Intake Agent 邮件、附件、上传文件、Webhook payload SourceDocument、接入分类、拒收/待审原因 不解析报告正文，不判断认可是否通过。
Document Understanding Agent SourceDocument、任务上下文、文档内容 DocumentEvidence、DocumentIssue、文档类型、字段、置信度、低置信提示 不推进任务状态，不直接发补充资料通知，不独自生成完整处置建议。
Approval Readiness Agent ApprovalTask、清单模板、全部证据、问题、规则和历史案例参考 ReadinessDecision、缺失项、冲突项、PERecommendation 不做最终审批，不替 PE 确认争议项。
Orchestrator Agent 事件、当前任务状态、人工确认结果 状态变更、ReviewItem、通知触发、补充请求 不调用 LLM，不执行文档理解。
Human Review Workbench ReviewItem、证据、问题、建议、历史案例参考、上下文 人工确认、修正、驳回、补充要求 不是自动 Agent，不做黑盒决策。

6. MVP 与后续版本边界

MVP

当前必须落地的能力

  • 统一资料接入和误拦恢复。
  • 文档证据化、问题识别和核心字段置信度。
  • 任务级认可就绪判断。
  • 轻量 PE 建议：规则优先，历史案例作为可选参考。
  • ReviewItem 人工复核闭环。
  • 事件日志和审计追踪。
V1.5+

后续扩展能力

  • Report Agent：认可报告草稿生成。
  • 独立 RAG Agent：历史案例召回与深度风险推理。
  • 供应商门户：供应商自助补充资料。
  • SFT：基于 PE 纠错样本做模型优化。
  • 复杂返工根因分析与供应商绩效看板。

7. 架构风险提示

不要把 Agent 按技术工具命名

Mail、Parser、Checker 容易把产品限制在技术实现上。Intake、Understanding、Readiness 更接近业务阶段。

不要让 DataChecker 只检查字段

OTS 认可真正关心的是任务是否就绪、问题是否阻塞、PE 下一步怎么处理，而不是某一份文档是否有几个字段。

不要绕开人工工作台

没有 ReviewItem 闭环，低置信和冲突只能散落在日志里，PE 无法稳定接管。

不要提前拆更多 Agent

MVP 不需要独立 Report Agent、RAG Agent、SFT Agent。RAG 可以作为建议参考能力，先把核心闭环打穿。

本架构图是 v3 产品设计视角，不要求当前工程代码立即完全对应；后续规格转让书应以此作为上游设计基线。
