OTS-AHA v3 Agent 链路图

v3 链路围绕 资料进入 → 文档证据化 → 问题识别 → 建议生成 → 人工确认/推进 设计。每个 Agent 只负责一个清晰阶段，输出可被下游消费的产品对象。

1. 主链路总览

资料来源 邮件、手动上传、Webhook、未来供应商门户。
Intake Agent 统一接入、过滤、去重、任务关联，生成 SourceDocument。
Document Understanding 文档类型识别、字段提取、问题识别，生成 DocumentEvidence 与 DocumentIssue。
Issue Map 聚合 FAIL、BORDERLINE、版本冲突、标准缺失、数值异常。
Recommendation 基于规则和历史案例，生成 PERecommendation。
Readiness 判断是否可会签、需补资料、需人审或被问题阻塞。
Orchestrator / Review 状态推进或创建人工确认项，PE 确认后回到主链路。

2. 关键链路原则

入口统一

邮件、上传、Webhook 都进入同一个 Intake 层，避免每个入口各自写一套过滤和关联逻辑。

证据优先

文档理解输出的是 DocumentEvidence，不是“同意/拒绝”的审批结论。

问题可追溯

每个 DocumentIssue 都必须能回到原文证据，不能只给出抽象判断。

任务级判断

认可就绪度必须以 ApprovalTask 为单位判断，而不是只看单份报告字段。

人审闭环

人工确认是主链路的一部分，不是异常旁路；所有人工结果都要回写并驱动状态推进。

3. 问题与建议泳道

文档理解
原文证据页码、表格、字段、截图区域。
问题识别FAIL、BORDERLINE、版本冲突、标准缺失。
置信度字段置信度 + 问题置信度。
输出DocumentEvidence + DocumentIssue
就绪判断
任务上下文零件、清单、供应商、项目阶段。
历史案例RAG 只提供参考案例与适用条件。
PE 建议补资料、复测、整改说明、风险会签。
输出ReadinessDecision + PERecommendation

4. 人工复核回路

v3 不把“需要人工”视为失败。相反，人工复核是 L3 级 AI Agent 的关键产品能力： 系统自动处理确定性部分，把不确定、冲突和业务风险集中交给 PE。 

flowchart TD
  Signal[低置信 / 缺失 / 冲突 / 建议动作 / 误拦恢复] --> CreateReview[Orchestrator 创建 ReviewItem]
  CreateReview --> Workbench[Human Review Workbench]
  Workbench --> Confirm[PE 确认无误]
  Workbench --> Correct[PE 修正字段或选择版本]
  Workbench --> AcceptRec[PE 采纳建议]
  Workbench --> Reject[PE 驳回或要求补充]

  Confirm --> Resume[继续自动推进]
  Correct --> EvidenceUpdate[更新 DocumentEvidence / Decision]
  AcceptRec --> Resume
  Reject --> Supplement[生成补充资料请求]
  EvidenceUpdate --> Readiness[重新执行认可就绪判断]
  Readiness --> Resume

6. 三个 PE 日常场景

场景 A

收到一封 OTS 报告邮件

Intake 识别为 OTS 资料并关联任务；文档理解生成测试结论、版本号、日期等证据，并标记测试问题。

场景 B

资料包缺少关键报告

Approval Readiness 输出 need_supplement，Orchestrator 创建补充资料请求，并在工作台展示缺失项。

场景 C

软件版本存在冲突

系统创建 ReviewItem，PE 在工作台确认有效版本；确认结果触发重新就绪判断。

场景 D

报告出现 FAIL 项

生成 DocumentIssue，结合历史案例生成 PERecommendation，由 PE 决定是否补资料、复测或进入风险会签。

不可接受链路

LLM 直接输出“通过审批”、功能 Agent 直接改任务状态、RAG 案例被当成自动审批依据，这三类行为在 v3 中必须禁止。
