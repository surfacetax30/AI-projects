# OTS 零部件认可 AI Agent 项目 — 评测方案框架

## 评测对象

**OTS-AHA (OTS Approval Helping Agent)** 全项目，包括以下子系统：

| 层级 | 子系统 | 核心组件 |
|------|--------|---------|
| 输入层 | 文件提取 + OCR | `extractor.py`（PDF/DOCX/XLSX/CSV/TXT/图片→文本） |
| 网关层 | 邮件过滤器 | `mail_gateway.py`（发件人白名单、主题关键词、附件校验） |
| 解析层 | LLM Parser | `parser.py` + `system_prompt.txt` → DeepSeek V4 结构化提取 |
| 校验层 | 数据完整性检查 | `data_checker.py`（基于零件类型 checklist 的完整性判定） |
| 总线层 | 事件驱动引擎 | `event_bus.py`（asyncio.Queue pub/sub，12 种事件类型） |
| 接口层 | REST API | 6 个端点（零件 CRUD / 任务查询 / 文件上传 / webhook） |
| 展示层 | 管理前端 | `index.html` 单页仪表板（5 模块） |
| 存储层 | 数据库 + 对象存储 | PostgreSQL（5 表）+ MinIO（已配置，待集成） |
| 运行层 | 脚本 + 测试 | `parse_file.py` CLI 工具、4 类测试（单元/集成/E2E/评测） |

## 五维度评测体系

### 维度 1：业务准确性 Business Accuracy — 权重 30%

> **衡量什么**：系统端到端能否正确提取 OTS 测试报告中的关键业务字段。这是 SQE 场景的底线——错误的 `part_no` 导致零件无法归档，错误的 `test_result` 可能导致不合格件放行。

#### MOS 1-5 打分规则

| MOS | 定义 | 核心字段准确率 | test_type 枚举 | test_result 映射 | 持续时长要求 |
|-----|------|:---:|:---:|:---:|------|
| **5** | 生产级精确，可作为自动化审批唯一依据 | ≥ 98% | 100% 正确 | 无误判 | 连续 200+ 真实报告无业务事故 |
| **4** | 高置信自动化，人工仅需抽查边界 | ≥ 95% | ≥ 98% | 无误判 | 连续 100+ 报告无业务事故 |
| **3** | 辅助决策可用，人工需逐份复核（**及格线**） | ≥ 90% | ≥ 95% | ≤ 1 次误判/50份 | 存在已知边界但不触发业务风险 |
| **2** | 仅作参考，不可用于决策辅助 | ≥ 80% | ≥ 85% | 存在≥2种已知误判模式 | 需人工逐字段校验 |
| **1** | 不可用，存在系统性错误 | < 80% | < 85% | 频繁误判 | 不适用 |

#### 评测方法
- 准备 **50 份带 ground truth 的真实 OTS 报告**（覆盖 6 种零件类型：金属支架/域控制器/电子零件/塑料件/软件件/EMC件）
- 逐字段比对提取值 vs 人工标注值
- 重点加权：`part_no`（3x）、`test_result`（3x）、`test_type`（2x）、`test_date`（2x）
- MOS 3 及格线：核心字段 90%+ 准确率

### 维度 2：流程完整性 Pipeline Completeness — 权重 25%

> **衡量什么**：从文件上传到最终判定结果的完整链路是否无断点，中间状态是否可追溯，事件是否完整发布。这是多人协作（Solo + Code）项目的核心质量指标——断点意味着交付不完整。

#### MOS 1-5 打分规则

| MOS | 定义 | 链路连续性 | 状态持久化 | 事件跟踪 | 回滚/重试 |
|-----|------|:---:|:---:|:---:|:---:|
| **5** | 全链路闭环，可生产部署 | 无断点 | 每步写入 DB | 12 种事件全部发布 | 支持重新处理 |
| **4** | 全链路跑通，已知缺口有文档记录 | 无关键断点 | 核心节点持久化 | ≥ 8 种事件发布 | 可手动触发重跑 |
| **3** | 核心链路跑通，存在不影响主流程的已知缺口（**及格线**） | 主流程不断 | 输入+输出持久化，中间状态在日志 | ≥ 4 种事件发布 | 可重新上传触发 |
| **2** | 链路存在关键断点，需人工介入才能完成 | 某一步失败后无法继续 | 仅部分输入持久化 | ≤ 2 种事件 | 不可重试 |
| **1** | 链路无法跑通 | 多个节点失败 | 无效持久化 | 无事件 | 不适用 |

#### 链路检查清单（通过/未通过/不适用）

| 检查项 | 对应代码位置 | 当前状态 |
|--------|-------------|---------|
| 文件上传 → 文本提取成功 | `tasks.py` → `extractor.py` | 待验证 |
| 文本提取 → REPORT_RECEIVED 事件发布 | `tasks.py` → `event_bus` | 待验证 |
| REPORT_RECEIVED → MailGateway 过滤 | `main.py` `_handle_report_received` | 待验证 |
| MailGateway 通过 → Parser LLM 调用 | `main.py` → `parser.py` | 待验证 |
| Parser 成功 → parsed 结果结构化 | `parser.py` | 待验证 |
| parsed 结果 → DataChecker 完整性校验 | `main.py` → `data_checker.py` | 待验证 |
| 全链路结果 → ParsedReport DB 写入 | `models.py` `ParsedReport` | **已知缺口**（表存在但未写入） |
| 任务状态 → ApprovalTask.state 更新 | `models.py` `ApprovalTask` | 待验证 |
| MinIO 文件实际存储 | `tasks.py` → MinIO | **已知缺口**（storage_path 生成但未上传） |
| Human Review 事件处理 | `event_bus` → 人审接口 | **已知缺口**（事件定义但无 handler） |

> MOS 3 及格线：以上检查项中「未通过」≤ 3 项，且所有已知缺口必须有文档记录（`规格转让书` 或 `设计文档` 中明确标注）。

### 维度 3：鲁棒性 Robustness — 权重 20%

> **衡量什么**：系统面对真实世界的多样性输入时，是优雅降级还是直接崩溃。真实 SQE 场景下，供应商会提交扫描件、手写件、甚至发错文件类型——系统不能只考标准卷。

#### MOS 1-5 打分规则

| MOS | 定义 | 文件格式覆盖 | OCR 可用性 | 异常处理 | 降级策略 |
|-----|------|:---:|:---:|:---:|:---:|
| **5** | 任何合理输入都有可预期的行为 | 全格式（含图片） | 中文+英文扫描件 ≥ 90% 可用 | 所有异常被捕获且有业务语义的 error message | 逐字段降级，部分失败≠全部失败 |
| **4** | 主要异常场景有兜底，边界场景有文档 | 5 种格式（PDF/DOCX/XLSX/CSV/TXT） | OCR ≥ 70% 可用 | 异常被捕获，错误信息可定位问题 | 核心字段失败时整体标记为 pending_human |
| **3** | 常见场景可用，异常能被捕获不崩溃（**及格线**） | 4 种格式（PDF/DOCX/XLSX/CSV） | OCR 存在但效果不稳定 | 不被意外输入击穿进程 | 异常输入返回 error 状态而非崩溃 |
| **2** | 仅标准格式可用，边界场景频繁出错 | 2-3 种格式 | OCR 不可用 | 部分异常导致 500 或进程退出 | 无降级，一次失败全部中止 |
| **1** | 单一格式可用，异常输入导致崩溃 | 1 种格式 | 无 | 频繁 500 或崩溃 | 无 |

#### 鲁棒性测试矩阵（12 项）

| # | 场景 | 预期行为 | 考察点 |
|---|------|---------|--------|
| 1 | 标准 PDF DV 报告（完整） | 正常提取，confidence ≥ 0.8 | 基准 |
| 2 | DOCX 软件版本报告 | 正常提取软件版本字段 | 格式 + 报告类型 |
| 3 | XLSX 多 sheet 报告 | 提取所有 sheet 文本 | 格式 |
| 4 | CSV/TSV 测试数据 | 自动识别分隔符 | 格式 |
| 5 | 扫描 PDF（无文字层） | 自动触发 OCR | OCR 能力 |
| 6 | 图片（PNG/JPG）报告 | OCR 提取 | OCR 能力 |
| 7 | 非测试报告（邮件正文） | overall=0.0, field_status=not_applicable | 拒绝判定 |
| 8 | 非测试报告（会议纪要） | overall=0.0, field_status=not_applicable | 拒绝判定 |
| 9 | 空文件（0 bytes） | 返回 error，不崩溃 | 异常处理 |
| 10 | 乱码/二进制文件 | 返回 error，不崩溃 | 异常处理 |
| 11 | 超大文件（> 50MB） | 超时/截断处理，不 OOM | 资源保护 |
| 12 | 并发上传 10 份报告 | 全部处理，结果不串 | 并发安全 |

> MOS 3 及格线：12 项中 ≥ 9 项行为符合预期。

### 维度 4：终端可用性 Usability — 权重 15%

> **衡量什么**：一个非技术背景的 SQE 工程师能否独立使用系统完成日常工作。这是产品交付的最终检验——代码写得再好，前端打不开等于没交付。

#### MOS 1-5 打分规则

| MOS | 定义 | 前端完备度 | API 可用性 | 错误信息质量 | 操作闭环 |
|-----|------|:---:|:---:|:---:|:---:|
| **5** | SQE 可独立完成全流程，无需技术支持 | 5 模块全部可用 | 6 端点全部正常 | 错误信息包含原因 + 下一步操作建议 | 上传→查看→确认/驳回→归档全闭环 |
| **4** | 核心流程可独立完成，高级功能需引导 | 3+ 模块可用 | API 正常响应 | 错误信息可定位问题 | 上传→查看→判定闭环 |
| **3** | 基本可操作，需要一定技术支持（**及格线**） | 至少 Dashboard + Upload 可用 | API 可被 curl/Postman 调用 | 错误信息至少包含 HTTP 状态码和简要描述 | 上传→看结果闭环 |
| **2** | 前端残缺，API 是唯一可用入口 | 前端不可用或严重 bug | API 部分端点可用 | 错误信息不明确 | 无闭环 |
| **1** | 无可用的用户界面 | 无前端或无法启动 | API 不可调用 | 无错误信息 | 无闭环 |

#### 可用性检查清单

| 检查项 | 验证方式 |
|--------|---------|
| 前端页面是否正常渲染（`GET /`） | 浏览器打开，5 个 Tab 可切换 |
| 零件创建流程是否完整可用 | 前端 → `POST /api/parts` → 返回 part + task |
| 文件上传是否可用（拖拽） | 前端拖拽 PDF → `POST /api/tasks/{id}/reports` → 返回 storage_path |
| 任务详情是否展示完整 | `GET /api/tasks/{id}` → 包含 part + reports + timeline |
| Webhook 模拟是否可用 | `POST /api/webhooks/mail` → 返回 accepted 状态 |
| 错误场景的前端提示是否友好 | 上传空文件/test_type 不匹配时前端显示什么 |
| API 是否能在没有前端时独立使用 | curl 调用全部 6 个端点 |
| 日志是否可追踪单个请求的全链路 | 按 task_id grep 日志 |

> MOS 3 及格线：8 项中 ≥ 6 项通过。

### 维度 5：性能与效率 Performance — 权重 10%

> **衡量什么**：系统在实际 SQE 工作流中的响应速度是否满足业务节奏。单份报告等 5 分钟，一天处理 20 份报告就不可接受。

#### MOS 1-5 打分规则

| MOS | 定义 | 单份报告端到端 | 并发处理 | 资源占用 | 扩展方案 |
|-----|------|:---:|:---:|:---:|:---:|
| **5** | 可处理批量审批，延迟无感知 | < 20s | 支持 5+ 并发 | CPU < 50%, 内存 < 2GB | 有队列扩展文档 |
| **4** | 日常使用流畅，偶有等待 | < 45s | 支持 3 并发 | CPU < 70%, 内存 < 4GB | 代码预留并发接口 |
| **3** | 可用，高峰期有感知延迟（**及格线**） | < 90s | 串行处理可用 | CPU < 85%, 内存 < 8GB | 文档标注性能瓶颈 |
| **2** | 有明显等待，不适合高频使用 | 90s - 3min | 串行处理有超时风险 | 偶发 OOM 或 CPU 100% | 无 |
| **1** | 不可用于生产 | > 3min 或频繁超时 | 不可并发 | 资源不足 | 无 |

#### 性能分项指标

| 环节 | 测量方法 | MOS 3 要求 | MOS 5 要求 |
|------|---------|:---:|:---:|
| 文件上传延迟 | 100KB - 10MB PDF 上传到 API 返回 | < 3s | < 1s |
| 文本提取用时 | `extract_text()` 耗时（不含 OCR） | < 5s | < 2s |
| OCR 处理用时 | 扫描件 OCR 全流程 | < 30s | < 15s |
| LLM 推理延迟 | API 调用到 JSON 返回（含 DeepSeek 服务端延迟） | < 60s | < 15s |
| 事件总线调度延迟 | `publish` → handler 执行 | < 500ms | < 100ms |
| DB 写入延迟 | 单条记录 insert | < 200ms | < 50ms |
| 端到端总延迟 | 上传请求 → Result 可查 | < 90s | < 20s |
| 内存占用（空闲） | uvicorn 进程 RSS | < 500MB | < 200MB |
| 内存占用（处理中） | 含 OCR 模型加载 | < 4GB | < 1.5GB |

#### 测试环境要求
- CPU: 4 核
- 内存: 8GB
- 网络: DeepSeek API 延迟 < 500ms（P50）
- 并发测试: 3 份报告同时上传

> MOS 3 及格线：端到端 < 90s，且无 OOM 或超时崩溃。

## 综合评分

### 加权公式

```
项目总分 = 业务准确性×0.30 + 流程完整性×0.25 + 鲁棒性×0.20 + 可用性×0.15 + 性能×0.10
```

### 等级映射

| 总分 | 等级 | 投产建议 |
|------|------|----------|
| ≥ 4.5 | **S** | 可投产，全维度优秀 |
| 4.0-4.4 | **A** | 可投产，个别维度持续优化 |
| 3.5-3.9 | **B** | 可灰度，2 个维度存在改善空间 |
| 3.0-3.4 | **C** | 需迭代，一个核心维度不达标 |
| 2.0-2.9 | **D** | 不可投产，多个核心维度不达标 |
| < 2.0 | **E** | 需重新设计架构 |

### 一票否决项

以下情况任一触发，项目直接判定为 **不可投产**，无论总分多少：

| 否决条件 | 对应维度 | 触发标准 |
|---------|---------|---------|
| 业务事故型错误 | 准确性 | 连续 20 份报告出现 `test_result` 映射错误（PASS ↔ FAIL 颠倒） |
| 数据丢失 | 流程完整性 | 上传成功的文件无法在系统中找到对应结果 |
| 进程崩溃 | 鲁棒性 | 任一合法输入导致 uvicorn 进程退出 |
| API 不可达 | 可用性 | API 连续 5 次请求中 ≥ 3 次返回 500 |
| 超时不可用 | 性能 | 单份报告处理超过 5 分钟 |

## 六、知识库运营方案

### 6.1 知识资产全景

项目当前包含 **7 类知识资产**，分布在代码、Prompt、文件目录和数据库中：

| # | 知识资产 | 当前位置 | 存储形式 | 对业务的影响 |
|---|---------|---------|---------|------------|
| 1 | 零件检查清单模板 | `data_checker.py` `DEFAULT_TEMPLATES` | Python dict 硬编码 | 错/漏 → 审批不完整 |
| 2 | Few-Shot 示例 | `system_prompt.txt` 示例1-4 | Prompt 内嵌文本 | 差 → LLM 输出质量下降 |
| 3 | 字段提取规则 + 特殊场景 | `system_prompt.txt` | Prompt 内嵌文本 | 错/漏 → 解析失真 |
| 4 | 供应商白名单 + 邮件过滤规则 | `mail_gateway.py` | Python 常量硬编码 | 漏 → 报告被误拦 |
| 5 | 测试报告样本集 | `real_OTS_data/`（22 文件） | 文件目录 | 无标注 → 无法自动评测 |
| 6 | 评测 ground truth | 无 | — | 无 → 准确性维度无法量化 |
| 7 | 零件类型分类体系 | `data_checker.py` + `system_prompt.txt` | 散落在代码和 SP 中 | 不一致 → 模块间分类冲突 |

### 6.2 资产一：零件检查清单模板

**业务含义**：每种零件类型（金属支架/域控制器/电子零件/塑料件）在 OTS 认可中需要提交哪些测试报告。这是 SQE 的核心业务规则。

| 维度 | 方案 |
|------|------|
| **数据从哪来** | SQE 团队提供的《零件 OTS 认可检查清单》纸质/电子版，由 **Solo（产品设计者）** 与 SQE 工程师逐项确认后录入 |
| **怎么入库** | ① 新增零件类型时，Solo 在 `产品设计文档/` 中输出 `规格转让书_零件检查清单_xxx.md`（含清单项 code/name/required）→ ② Code 通过管理 API 写入 `ChecklistTemplate` 表（DB 模型已存在但未使用）→ ③ `DataChecker` 改为读取 DB 而非硬编码 dict |
| **更新周期** | 随 OEM 认可标准变更触发（不定期，预计 2-4 次/年）；每年 Q1 做一次全量审计（核对 DB 清单 vs SQE 最新要求） |
| **负责人** | **Solo** 负责内容准确性（与 SQE 对齐），**Code** 负责入库脚本和 API |
| **版本管理** | `ChecklistTemplate.version` 字段自增；每次修改旧版本 `is_active=false`，新版本 `is_active=true`；保留历史版本用于回溯已审批的历史任务 |

**当前缺口**：`ChecklistTemplate` 表已建但 **DataChecker 仍读硬编码 dict**。需 Code 完成 DB 读取迁移。

### 6.3 资产二：Few-Shot 示例

**业务含义**：嵌入 System Prompt 的 4 个示例（标准 DV 报告 / 软件版本报告 / 简略手写报告 / 非测试报告），是 LLM 行为校准的主要手段。

| 维度 | 方案 |
|------|------|
| **数据从哪来** | ① 从 `real_OTS_data/` 中筛选典型报告作为素材 → ② Solo 人工标注 ground truth → ③ 编入 system_prompt.txt 的 Few-Shot 区块 |
| **怎么入库** | Solo 直接编辑 `system_prompt.txt`（当前唯一入口）；后续可设计独立 `few_shot_examples.json` 文件，由 `parser.py` 启动时动态拼入 prompt |
| **更新周期** | **按需触发**：评测发现 LLM 在特定场景持续出错时，新增针对性 Few-Shot；**季度评估**：检查是否有冗余/过时示例需要移除或替换 |
| **负责人** | **Solo** 全权负责（内容设计 + prompt 测试） |
| **版本管理** | `system_prompt.txt` 纳入 Git；Few-Shot 变更需在 `SystemPrompt评测报告.md` 中记录变更原因和效果对比 |
| **数量上限** | 建议控制在 5-7 例以内（超过 7 例会挤占 tokens，且可能造成 LLM 混淆） |

**当前状态**：4 例，覆盖机械件 / 软件件 / 低质量 / 拒绝 四类，暂无需新增。

### 6.4 资产三：字段提取规则 + 特殊场景

**业务含义**：System Prompt 中的核心——15 字段提取规则表、5 条特殊场景规则、置信度评分指南、field_status 三态规则、硬性降分条件。

| 维度 | 方案 |
|------|------|
| **数据从哪来** | ① SQE 业务需求（如新增 `software_version` 三个字段）→ ② Solo 理解业务后设计规则 → ③ 写入 `system_prompt.txt` → ④ 跑评测验证 |
| **怎么入库** | 与 Few-Shot 相同：Solo 编辑 `system_prompt.txt`，同步更新 `parser.py` 中 `SYSTEM_PROMPT` 常量（通过 `规格转让书` 交给 Code） |
| **更新周期** | **业务驱动**：OEM 认可标准变化、新零件类型上线时更新；**反馈驱动**：评测发现规则漏洞时修补 |
| **负责人** | **Solo** 负责规则设计和测试验证，**Code** 负责同步 `parser.py` |
| **关键风险** | `system_prompt.txt` 和 `parser.py` 中 `SYSTEM_PROMPT` 常量可能不一致——这是当前最大的知识漂移风险。脚本 `parse_file.py` 读 `.txt`，API 管线 `parser.py` 读自身常量。**必须建立同步 Check List** |

**当前缺口**：`system_prompt.txt` 已更新 `field_status` + 加权置信度，但 `parser.py` 中 `SYSTEM_PROMPT` 仍是上一个版本。

### 6.5 资产四：供应商白名单 + 邮件过滤规则

**业务含义**：`MailGateway` 的判断依据——哪些发件人可信、哪些主题关键词放行、哪些附件后缀接受。

| 维度 | 方案 |
|------|------|
| **数据从哪来** | ① SQE 团队维护的《合格供应商名录》→ ② Solo 确认后录入 |
| **怎么入库** | 短期：通过配置文件（`config/suppliers.yaml` 或 `.env` 中的逗号分隔列表）；长期：通过管理 API + 前端供应商管理页面写入 DB 表 `supplier_whitelist` |
| **更新周期** | 新供应商准入时即时更新（预计 1-2 次/月）；每半年做一次全量审计 |
| **负责人** | **Solo** 确认供应商资质，**Code** 负责配置/DB 写入 |
| **风险** | 如果白名单漏了某供应商，其提交的报告会被 `MailGateway` 静默拦截——需要日志告警 + `REPORT_RECEIVED` 事件中记录 reject reason |

**当前状态**：3 个示例邮箱硬编码在 `mail_gateway.py`，需要迁移到配置文件或 DB。

### 6.6 资产五 & 六：测试报告样本集 + Ground Truth

### 6.7 资产七：零件类型分类体系

**业务含义**：不同模块（`DataChecker` 的 `part_type`、`system_prompt.txt` 的 `test_type` 枚举、`ChecklistTemplate` 表）都在引用零件分类，但定义分散，存在不一致风险。

| 维度 | 方案 |
|------|------|
| **数据从哪来** | SQE 团队维护的《零件分类编码表》 |
| **怎么入库** | 集中到一个 `config/part_taxonomy.yaml` 文件中，定义统一的分类编码和模块映射关系 |
| **更新周期** | 与资产一（检查清单）联动更新，同一触发条件 |
| **负责人** | **Solo** 设计分类体系并保证一致性，**Code** 确保所有模块读同一份配置 |

**分类体系建议结构**（`config/part_taxonomy.yaml`）：

### 6.9 知识库同步 Check List（每次 SP 变更后必做）

当前最大的知识管理风险是 `system_prompt.txt` 和 `parser.py` 中的 `SYSTEM_PROMPT` 常量不同步。建议每次 SP 变更后执行以下检查：

| # | 检查项 | 验证方式 |
|---|--------|---------|
| 1 | `system_prompt.txt` 最新修改时间 < 24h，`parser.py` 中 `SYSTEM_PROMPT` 是否一致？ | diff 对比 |
| 2 | `parse_file.py` 测试 2 份报告（软件报告 + DV 报告），结果是否符合预期？ | 运行脚本 |
| 3 | `test_system_prompt_eval.py` 5 个评测用例是否全部通过？ | 运行 pytest |
| 4 | 新增的字段/规则/场景是否同步更新了 `规格转让书` 文档？ | 检查 `产品设计文档/` 目录 |
| 5 | `DataChecker` 的 `DEFAULT_TEMPLATES` 是否需要同步更新？ | 按零件类型逐项核对 |

### 6.10 知识库运营对评测维度的影响

知识库质量直接影响评测框架的 5 个维度：

| 评测维度 | 知识库依赖 | 恶化信号 |
|---------|-----------|---------|
| 业务准确性 | Few-Shot 质量 + 提取规则完整性 | 评测准确率连续 2 次下降 |
| 流程完整性 | 检查清单模板是否入库 | DataChecker 仍读硬编码 |
| 鲁棒性 | 样本集是否覆盖所有零件类型 | 某类型报告无标注数据 |
| 可用性 | 白名单是否更新 | 供应商反馈报告被误拦 |
| 性能 | （不直接依赖知识库） | — |
