OTS-AHA 评测方案 v3.0

从 Parser/SP 评测升级为 v3 Agent MVP 核心链路评测

评测对象Intake / Document Understanding / Approval Readiness / Human Review Workbench
修订依据在 v2.0 Parser/SP 评测基础上，补充问题识别、PE建议、就绪判断
核心原则真实报告优先；评测先于优化；建议不替代 PE 决策
产出目标MVP 阶段建立最小可信评测集，并跑出第一版基线分
重点结论 v2.0 的指标设计适合作为 Parser/SP 单点评测，但已经不足以衡量 v3 MVP 的产品目标。v3.0 必须覆盖 资料接入、文档证据化、问题识别、PE建议、任务级认可就绪判断。

1. 评测对象与核心目标

v3 MVP 的核心目标不是单纯“把报告字段解析出来”，而是帮助 PE 每天处理 OTS 认可资料包：系统接入资料、理解文档、识别问题、提出建议，并把不确定事项交给 PE 形成闭环。

30%

继承 Parser/SP 基本盘，评估核心字段、证据出处和文档类型。

20%

判断哪些结果能自动使用，哪些必须交给 PE。

验证系统是否能看出测试文档中的业务问题。

20%

判断资料包能否推进会签、补料、人审或阻断。

10%

评估建议是否有依据、可执行、不越权。

评测维度为什么重要MVP 建议达标线
文档证据准确性这是所有后续判断的地基。核心字段准确率 ≥90%；test_result 不允许高风险反向错误。
置信度与状态校准决定自动推进和人工确认边界。校准偏差 ≤0.20；field_status 判断准确率 ≥90%。
测试问题识别能力从“读懂字段”升级到“看出问题”。关键问题召回率 ≥85%；严重等级判断准确率 ≥75%。
认可就绪判断能力对应 PE 是否能知道资料包能不能推进。任务级判断准确率 ≥80%；阻断型问题漏判为 0。
PE 建议质量体现 Agent 对 PE 的辅助价值。建议被采纳或小改后采纳率 ≥60%；100% 给出依据。
门禁而非加分 JSON/Schema 输出规范性不再作为 15% 权重项，而应作为 Gate：格式不合格则该 case 直接不可用或进入修复。

3. 分维度指标定义

3.1 DocumentEvidence：文档证据准确性

指标定义阈值
核心字段准确率part_no、supplier、test_item、test_result、test_standard、software_version 等核心字段逐项比对 GT。≥90%
证据出处命中率每个关键字段必须能回指页码、表格、段落或单元格。≥85%
文档类型识别准确率区分测试报告、软件版本报告、补充说明、非 OTS 资料。≥90%

3.2 置信度与状态校准

指标定义阈值
置信度校准偏差预测置信度与实际正确率之间的偏差。≤0.20
field_status 准确率correct / ambiguous / missing / conflict / low_confidence 的判断准确率。≥90%
高置信误判率confidence ≥0.85 但结果错误的比例。≤5%
低置信召回率真实不确定、缺失、冲突字段被标为需确认的比例。≥85%

3.3 DocumentIssue：测试问题识别能力

问题类型示例输出要求
test_failed测试项 FAIL、NG、不满足标准。指出问题字段、原文证据、影响范围。
conditional_pass有条件通过、限用、需补充验证。不能简单归为 PASS，需进入人审或补料。
version_conflict报告软件版本与任务/其他文档版本不一致。指出冲突双方及来源。
standard_missing缺少测试标准、判定依据、限值。进入 need_supplement 或 need_human_review。
value_anomaly数值异常、单位不一致、边界值临界。给出异常原因与置信度。
指标定义阈值
关键问题召回率GT 标注为严重/阻断的问题被识别出来。≥85%
问题类型准确率DocumentIssue.type 与 GT 一致。≥80%
严重等级准确率blocking / warning / info 与 GT 一致。≥75%
高风险漏判阻断型 FAIL、版本冲突被判断为无问题。0 容忍

3.4 ReadinessDecision：认可就绪判断

状态含义典型触发条件
ready_for_signing资料包可进入会签。核心资料齐全，无阻断问题，无待确认高风险项。
need_supplement需要补充资料。缺少测试报告、标准、版本说明、签章页等必要材料。
need_human_review需要 PE 判断。存在低置信、边界通过、口径不一致、建议需要人工确认。
blocked_by_conflict被冲突或严重问题阻断。FAIL、版本冲突、关键标准缺失、证据互相矛盾。

3.5 PERecommendation：PE 建议质量

指标定义阈值
建议动作准确率建议补料、复测、确认版本、推进会签等动作是否合理。≥70%
建议采纳率PE 直接采纳或小改后采纳的比例。≥60%
依据完整率建议必须引用 DocumentEvidence、DocumentIssue 或历史案例来源。100%
越权率建议不能替 PE 做最终审批，不能把 RAG 案例当审批依据。0%
历史案例适配率引用历史案例与当前问题类型、物料、测试项相似。≥80%
PERecommendation 展示规则 每条建议必须包含：建议动作、理由、证据引用、历史案例参考、是否需要 PE 确认。RAG 的定位是“经验参考”，不是“自动审批依据”。

5. MVP 评测集设计

子集建议数量覆盖内容用途
A. Parser/SP 字段抽取集沿用并清洗现有 26 个 casePDF/DOCX/XLSX；核心字段、置信度、field_status保留 v2.0 的基本盘
B. Issue Detection 问题识别集8-12 个 caseFAIL、边界通过、版本冲突、标准缺失、数值异常验证文档理解是否真正理解问题
C. Readiness 资料包就绪集6-8 个任务包每个任务包含 2-5 份资料，覆盖缺料/冲突/可会签验证任务级判断
D. Recommendation 建议集5-8 个 case补料建议、复测建议、版本确认建议、会签推进建议验证建议是否能帮 PE 省时间
E. Intake/Negative 接入与负样本集8-10 个 case非 OTS 邮件/附件、重复资料、错发资料、弱相关资料避免垃圾资料进入后续链路
MVP 完成标准 评测集建立完成不是指样本数量很多，而是每个核心能力都有 GT、覆盖矩阵、可运行脚本、基线分和错误归因方式。

7. 评测流程与角色分工

1. 升级标注指南新增 DocumentIssue、ReadinessDecision、PERecommendation 的定义和示例。
2. 清洗现有 Parser/SP 集统一字段命名、置信度口径和 evidence_ref。
3. 补充问题识别样本优先从真实失败、边界通过、版本冲突报告中抽样。
4. 补充资料包就绪样本按“一个 OTS 认可任务”组织多文档 case。
5. 补充建议样本让 PE 标注理想建议动作和不可接受建议边界。
6. 执行基线评测输出整体分、分维度分、一票否决列表和 Top 错误原因。
