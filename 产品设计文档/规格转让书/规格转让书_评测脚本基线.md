## 三、运行流程

```
Step 1: 加载 eval_cases.json
Step 2: 初始化 LLMClient（复用 app/services/llm.py）
Step 3: 对每条 case：
  a. 调 DeepSeek V4 → system_prompt = load_sp("parser"), user_message = input
  b. 提取 ```json ``` 代码块 → json.loads()
  c. 如果 JSON 解析失败 → 记录 parse_error=true，跳过字段比对
  d. 逐字段比对 actual vs expected（见第四章）
  e. 记录每条 case 的比对结果
Step 4: 汇总计算 4 维度得分（见第五章）
Step 5: 检查 7 项一票否决（见第六章）
Step 6: 输出评测报告
```

## 四、逐字段比对规则

对每条 case 的 15 个业务字段 + field_confidence + field_status + overall_confidence，做以下 7 项比对：

| # | 比对项 | 规则 | 计入维度 |
|---|--------|------|---------|
| 1 | **字段值匹配** | 核心字段（part_no/test_result/test_type/test_date）2x 权重；其余 1x | 准确性 |
| 2 | **part_no 比对** | 忽略大小写；纯数字原样保留 | 准确性 |
| 3 | **test_date 比对** | 统一规范化为 YYYY-MM-DD 后比较 | 准确性 |
| 4 | **material 比对** | 牌号比对，允许等价表述（"45#钢" = "45钢" = "45号钢"） | 准确性 |
| 5 | **null 比对** | 期望 null 且实际 null → 命中；期望 null 但实际有值 → 未命中 | 准确性 |
| 6 | **field_status 比对** | extracted/missing/not_applicable 三态精确匹配 | 置信度校准 |
| 7 | **confidence 比对** | field_confidence 不在校准维度做区间判断（见下文），只记录偏差值 | 置信度校准 |

### 4.1 material 等价映射表（Code 硬编码）

```python
MATERIAL_EQUIV = {
    "45钢": "45#钢", "45号钢": "45#钢",
    "45#钢": "45#钢",
    # 可扩展
}
```

### 4.2 field_confidence 比对

不要求精确匹配 expected 中的 confidence 值。改为：对每个 extracted 字段，记录 `|predicted_confidence - actual_correctness|`：

- 字段值匹配正确 → `actual_correctness = 1.0`
- 字段值匹配错误 → `actual_correctness = 0.0`

所有 extracted 字段的偏差取均值 = **置信度偏差**（维度二的子指标）。

## 五、4 维度评分计算

### 维度一：字段提取准确性（权重 35%）

| 子指标 | 计算公式 | 占比 |
|--------|---------|:---:|
| 核心字段准确率 | 核心字段命中数 / 核心字段总数 | 50% |
| 一般字段准确率 | 一般字段命中数 / 一般字段总数 | 30% |
| test_result 映射 | PASS/FAIL/CONDITIONAL 枚举正确率 | 10% |
| test_type 枚举匹配 | 17 种枚举 + "其他" 正确率 | 10% |

```
核心字段 = [part_no, test_result, test_type, test_date]
加权命中 = 核心字段命中×2 + 一般字段命中×1
加权总数 = 核心字段数×2 + 一般字段数×1
加权准确率 = 加权命中 / 加权总数

维度一得分 = MOS(加权准确率)  ← 查 MOS 表（≥98%→5, ≥95%→4, ≥90%→3, ≥80%→2, <80%→1）
```

### 维度二：置信度校准质量（权重 25%）

| 子指标 | 计算公式 | MOS 3 及格线 | MOS 5 满分线 |
|--------|---------|:---:|:---:|
| 置信度偏差 | mean(\|pred_conf - actual_correctness\|) | ≤ 0.20 | ≤ 0.05 |
| field_status 准确率 | 三态判定正确数 / 总数 | ≥ 90% | ≥ 98% |
| 校准一致性 | 高置信度(≥0.8)字段的实际准确率 | ≥ 90% | 100% |
| 误报率 | extracted 但值错误的比率 | ≤ 10% | ≤ 2% |

```
维度二得分 = mean(四个子指标的 MOS 映射值)
```

### 维度三：边界场景鲁棒性（权重 25%）

| 子指标 | 计算 | MOS 3 及格 |
|--------|------|:---:|
| 边界 case 通过率 | (C11-C16 中通过数) / 6 | ≥ 75% |
| 非报告拒绝率 | (C17-C21 中 overall=0.0 数) / 5 | ≥ 90% |

> 边界 case 通过定义：核心字段准确率 ≥ 80% 且 test_result 无颠倒。

```
维度三得分 = MOS(边界通过率×0.6 + 非报告拒绝率×0.4 后查 MOS 表)
```

### 维度四：输出规范性（权重 15%）

对 21 条 case 统计以下 10 项检查（见 PDF 第六章）：

| 检查项 | 判定 |
|--------|------|
| JSON 可解析 | `json.loads()` 无异常 |
| 15 个业务字段齐全 | 全部 key 存在 |
| field_confidence 齐全 | 15 个 key，值在 [0,1] |
| field_status 齐全 | 15 个 key，值在三态枚举中 |
| overall_confidence 存在 | 在 [0,1] |
| notes 存在 | 非空 |
| test_result 枚举合规 | PASS/FAIL/CONDITIONAL/null |
| test_type 枚举合规 | 17种 + "其他" |
| test_date 格式合规 | YYYY-MM-DD 或 null |
| 无多余字段 | 不超过 Schema 定义的 key |

```
Schema 合规率 = 通过检查项总数 / (10 × 21 case)
维度四得分 = MOS(Schema合规率)  ← ≥95%→3, ≥98%→4, 100%→5
```

### 综合评分

```
总分 = 维度一×0.35 + 维度二×0.25 + 维度三×0.25 + 维度四×0.15
```

## 六、一票否决检查（7 项）

在 21 条 case 上检查以下条件，任意触发则标记 `VETO`：

| # | 否决条件 | 检查方式 |
|---|---------|---------|
| V1 | test_result 颠倒 | 任意 case 中 expected=PASS 但 actual=FAIL（或反之） |
| V2 | JSON 大面积无法解析 | ≥ 3 条 case `json.loads()` 失败 |
| V3 | 非报告误判为报告 | C17/C18/C19 中 overall_confidence > 0.3 |
| V4 | 软件报告缺版本字段 | C05 中 software_version_* 全为占位文本而非提取值 |
| V5 | CONDITIONAL 误映射为 PASS | C13 中 actual test_result = "PASS"（期望 CONDITIONAL） |
| V6 | 置信度虚高 | ≥ 5 条 case 中提取错误字段的 confidence > 0.8 |
| V7 | field_status 系统性错误 | 非软件报告的 software_version 标为 extracted，或软件报告的 material 标为 extracted |
